# 前端登录鉴权对接指南

本文档说明前端如何与后端登录鉴权链路对接。后端采用 **JWT Bearer Token** 方案,所有受保护接口都需要在请求头携带 token。

---

## 一、核心约定(最重要)

### 1. Token 传输方式

后端生成的 token 是**裸 JWT**(三段式 `header.payload.signature`),**不带** `Bearer ` 前缀。

前端在请求受保护接口时,必须按 HTTP 标准格式拼接前缀后放入请求头:

```
Authorization: Bearer <裸token>
```

- 请求头名固定为 **`Authorization`**(不是 `Bearer`、不是 `token`)
- 值必须以 `Bearer ` 开头(注意 `Bearer` 和 token 之间有一个空格)
- `Bearer ` 前缀由前端拼接,**不要**把前缀存进 token 本身

### 2. 鉴权失败的响应特征

后端鉴权失败时 **HTTP 状态码仍是 200**,真正的错误信息在响应体的 `code` 字段里。前端必须按 `code` 判断,不能只看 HTTP 状态码。

---

## 二、接口清单

基路径:`/api/v1/auth`

| 方法 | 路径 | 是否需要 token | 说明 |
|------|------|---------------|------|
| POST | `/api/v1/auth/login` | ❌ 否 | 登录,返回 JWT |
| POST | `/api/v1/auth/logout` | ✅ 是 | 登出,token 立即失效 |
| GET  | `/api/v1/auth/me` | ✅ 是 | 获取当前登录用户信息 |
| PUT  | `/api/v1/auth/password` | ✅ 是 | 修改密码 |

除 `/login` 外,所有 `/api/**` 接口都需要 token。

---

## 三、登录接口

### 请求

```
POST /api/v1/auth/login
Content-Type: application/json
```

请求体:

```json
{
  "id": 1,
  "password": "123456"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | Integer | 是 | 员工工号(employee.id) |
| `password` | String | 是 | 密码(明文,后端负责加密校验) |

### 成功响应

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9...",
    "expiresAt": 1782216675000,
    "userInfo": {
      "id": 1,
      "realname": "李医生",
      "deptmentId": 1,
      "deptName": "神经内科",
      "registLevelId": 1,
      "registLevelName": "普通号"
    }
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `data.token` | String | 裸 JWT,前端需持久化保存 |
| `data.expiresAt` | Long | 过期时间戳(毫秒),可用于前端提前刷新/提示 |
| `data.userInfo` | Object | 当前登录用户基本信息 |

`userInfo` 字段:

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Integer | 员工ID |
| `realname` | String | 真实姓名 |
| `deptmentId` | Integer | 所在科室ID |
| `deptName` | String | 所在科室名称 |
| `registLevelId` | Integer | 挂号级别ID |
| `registLevelName` | String | 挂号级别名称 |

### 失败响应

```json
{
  "code": 401,
  "message": "工号或密码错误",
  "data": null
}
```

可能的错误码:

| code | message | 触发场景 |
|------|---------|---------|
| 401 | 工号或密码错误 | 工号不存在或密码错误 |
| 423 | 账号已锁定,请稍后再试 | 连续失败超过 5 次,锁定 15 分钟 |

> **注意**:连续登录失败 5 次会锁定账号 15 分钟,锁定期间即使密码正确也会返回 423。前端可据 `code=423` 提示用户等待。

## 四、携带 Token 请求(受保护接口通用)

登录成功后,所有受保护接口都需在请求头携带 token。以获取当前用户信息为例:

```
GET /api/v1/auth/me
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

### 鉴权失败错误码

| code | message | 含义 | 前端处理建议 |
|------|---------|------|------------|
| 401 | 未登录 | 未携带 token | 跳转登录页 |
| 401 | 令牌已过期,请重新登录 | token 过期(8 小时) | 清除本地 token,跳转登录页 |
| 401 | 令牌无效 | token 被篡改/解析失败,或已登出被加入黑名单 | 清除本地 token,跳转登录页 |

> **统一处理建议**:在 axios/fetch 拦截器里,若响应 `code` 为 401(任意 message),清除本地存储的 token 并跳转登录页。

---

## 五、登出接口

```
POST /api/v1/auth/logout
Authorization: Bearer <token>
```

无请求体。后端会将该 token 加入 Redis 黑名单,**立即失效**。

响应:

```json
{ "code": 200, "message": "操作成功", "data": null }
```

登出后前端应清除本地 token。此后该 token 再请求任何接口都会返回 `401 令牌无效`。

---

## 六、修改密码接口

```
PUT /api/v1/auth/password
Authorization: Bearer <token>
Content-Type: application/json
```

请求体:

```json
{
  "oldPassword": "123456",
  "newPassword": "abcdef"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `oldPassword` | String | 是 | 原密码 |
| `newPassword` | String | 是 | 新密码,**至少 6 位** |

错误码:

| code | message | 场景 |
|------|---------|------|
| 400 | 密码长度至少6位 | 新密码不足 6 位 |
| 401 | 工号或密码错误 | 原密码错误 |

> 修改密码后**旧 token 仍然有效**(后端未强制下线),如需立即失效可调用 `/logout`。

## 七、前端实现参考

### 1. 存储 token

登录成功后持久化保存(推荐 `localStorage`,刷新页面不丢失):

```js
const { token, expiresAt, userInfo } = res.data
localStorage.setItem('token', token)
localStorage.setItem('expiresAt', String(expiresAt))
localStorage.setItem('userInfo', JSON.stringify(userInfo))
```

### 2. axios 请求拦截器:自动拼接 Authorization

```js
import axios from 'axios'

const http = axios.create({ baseURL: '/api' })

http.interceptors.request.use(config => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`   // 注意前缀和空格
  }
  return config
})
```

### 3. axios 响应拦截器:统一处理鉴权失败

```js
http.interceptors.response.use(
  res => {
    const { code, message } = res.data
    if (code === 401) {
      // 未登录 / token 过期 / token 无效
      localStorage.removeItem('token')
      localStorage.removeItem('userInfo')
      router.push('/login')
      return Promise.reject(new Error(message))
    }
    if (code === 423) {
      // 账号锁定,提示用户
      return Promise.reject(new Error(message))
    }
    return res.data
  },
  error => Promise.reject(error)
)
```

### 4. 登录前置守卫(路由)

```js
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  if (to.meta.requiresAuth && !token) {
    next('/login')
  } else {
    next()
  }
})
```

## 八、常见对接陷阱

1. **请求头名写错**:必须是 `Authorization`,不是 `Bearer`、`token`、`Authorization Token`。
2. **漏掉 `Bearer ` 前缀或空格**:值必须是 `Bearer <token>`,缺前缀或缺空格都会导致后端读不到 token,返回 `401 未登录`。
3. **不要给 token 自身加前缀再存**:存的时候存裸 token,发送时再拼前缀,两步分离。
4. **只看 HTTP 状态码**:后端鉴权失败 HTTP 仍是 200,必须检查响应体 `code`。
5. **token 过期时间 8 小时**:`expiresAt` 是毫秒时间戳,可据此在临近过期时提示用户重新登录(本项目无刷新 token 机制)。
6. **登出后旧 token 立即失效**:登出后必须清除本地 token,不要继续使用。
7. **密码修改后 token 仍有效**:如需立即重新登录,改完密码后主动调 `/logout`。
